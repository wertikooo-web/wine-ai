Итоговый анализ: что сделано, что работает, что осталось. Живой сайт: https://wine-ai-realtime-production.up.railway.app/dashboard · Репозиторий: https://github.com/wertikooo-web/wine-ai

## 1. Что полностью готово и работает

### Realtime-ядро
Перенесено из проверенного `lunara-realtime` (независимо, без runtime-зависимости — см. `docs/WINE_AI_MIGRATION_PLAN.md`): WebSocket-протокол, turn/generation state machine, PCM16-ресемплер, ротация провайдера, таймауты/ретраи. 6/6 автотестов проходят, включая полный цикл прерывания/barge-in/reconnect.

### Голос и текст, 9 языков
Push-to-talk + текстовый ввод. Автоопределение языка (ru/ro/en/fr/it/es/de/zh/ja) плюс явный выбор языка в дашборде (dropdown). Детектор языка исправлен от коллизии винных когнатов ("vin"/"vino" путали fr/ro/it/es между собой).

### Персона
Финальный промпт от пользователя, с явными правилами: факт vs мнение vs рекомендация, никогда не выдумывать производителей/награды/цены/винтажи, честно говорить «нет подтверждённых данных», не давать категоричных медицинских утверждений об алкоголе, не поощрять злоупотребление.

### База знаний
- 6 вручную написанных документов + 22 реально собранных краулером (wineofmoldova.com/ONVV на 3 языках, Purcari Wineries, Cricova) — все источники проверены браузером перед добавлением, ни один URL не выдуман.
- **Загрузка файлов через дашборд** (drag & drop) — работает, проверено реальным браузерным событием.
- **Постоянное Postgres-хранилище** очереди документов (approved/pending/rejected) на Railway — с файловым fallback для локальной разработки.
- Поиск: keyword/term-overlap (без embeddings), с исправленным ранжированием (стоп-слова исключены из скоринга, метаданные весят больше сырого совпадения в тексте).

### 6 инструментов (function calling)
`search_wine_knowledge`, `search_winery`, `compare_grape_varieties`, `recommend_wine_pairing`, `get_wine_route_information`, `update_session_memory`.

### Дашборд
Live avatar (голос+текст+выбор языка+фото аватара), Knowledge base (статус, drag&drop загрузка, ручной запуск краулера, Knowledge Monitor с approve/reject), Persona, Diagnostics.

### Инфраструктура
- GitHub: `wertikooo-web/wine-ai`, ветка `main`, вся история коммитов честная и содержательная.
- Railway: сервис `wine-ai-realtime` + Postgres addon, автодеплой через CLI, `GEMINI_API_KEY` установлен пользователем напрямую (ассистент никогда не вводил/не печатал секреты).
- Тесты: 6 unit/integration + 4 smoke-скрипта, все зелёные перед каждым деплоем.

## 2. Баги, найденные и исправленные в процессе

Честный список — то, что казалось готовым, но не работало, пока не проверили вживую:

1. **Прерывание не работало** — сервер всё делал правильно, но дашборд ждал ответа сервера прежде чем остановить уже играющий звук. Исправлено: мгновенная остановка на клиенте при нажатии кнопки, не дожидаясь сети.
2. **Микрофон «не слышал»** — вероятная причина: WebRTC-обработка звука (echoCancellation/noiseSuppression/autoGainControl) иногда обнуляет обычную речь как «шум», плюс жёстко прописанная частота дискретизации 16000 могла не совпадать с реальной частотой браузера. Оба исправлены.
3. **Реальный краш всего процесса в проде** — Postgres возвращает даты как объекты `Date`, не строки; один роут упал на `.localeCompare()`, что уронило весь процесс вместе со всеми активными разговорами. Исправлено + добавлена защита (try/catch на весь HTTP-обработчик, process-level перехватчики).
4. **Фоновый (detached) процесс обновления базы работал невидимо** — молча ничего не делал, ошибки нигде не показывались. Переписано на синхронный вызов в том же процессе.
5. **Коллизия винных слов в определении языка** — "vin"/"vino" одинаково значат «вино» в нескольких романских языках, что вызывало вечную неоднозначность именно в винных разговорах. Убраны из различающих списков слов.
6. **Регрессия ранжирования поиска** — после роста базы длинный общий текст стал иногда обгонять короткий целевой документ. Поправлено.

## 3. Что НЕ сделано (осознанно, для следующих итераций)

| Что | Почему не сейчас |
| --- | --- |
| Реальный аватар (видео/анимация) | Явно отложено по вашей просьбе — «добавим позже» |
| Firecrawl / автопоиск по интернету (discovery) | Требует стороннего API-ключа, который должен завести сам пользователь |
| Vector DB (Qdrant) + embeddings | Осознанно отложено — сначала нужна реальная нагрузка, чтобы оправдать сложность/стоимость; текущий keyword-поиск работает достаточно хорошо для размера базы |
| Остальные сайты виноделен (Castel Mimi, Mileștii Mici, Asconi, Chateau Vartely и т.д.) | Каждый URL нужно так же проверить браузером перед добавлением — сделано только для 2 из ~10+ известных виноделен |
| Cron-расписание на Railway | CLI не поддерживает настройку через терминал — единственный оставшийся ручной шаг: Railway → сервис → Settings → Cron Schedule → `0 2 * * *` (скрипт сам проверяет, прошло ли 72 часа) |
| Открытый микрофон (VAD) вместо push-to-talk | Осознанный выбор v1 — см. `docs/WINE_AI_MIGRATION_PLAN.md` §5 |
| Восстановление сессии при обрыве связи | Каждое новое подключение — новая сессия с нуля, контекст разговора не сохраняется |
| Постоянная память между визитами/пользователями | Только в рамках одной сессии, ничего не сохраняется между посещениями |
| Извлечение фактов на уровне полей (цены, награды, владельцы) | Краулер сохраняет документ целиком; структурированное извлечение конкретных фактов с проверкой — следующий этап |

## 4. Рекомендации на следующий шаг (не сделано, только предложение)

1. Настроить Cron Schedule на Railway вручную (2 минуты в дашборде Railway).
2. Проверить и добавить ещё 5-10 сайтов виноделен в реестр источников.
3. Если база вырастет значительно — рассмотреть переход на embeddings/Qdrant (архитектура уже к этому готова, см. `docs/KNOWLEDGE_PIPELINE_ARCHITECTURE.md` §8).
4. Реальный аватар — когда решите вернуться к этому.
