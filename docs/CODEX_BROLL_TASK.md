# Задание для Codex: анализ и план внедрения динамического B-roll в WINE AI

## Контекст проекта

Проект WINE AI представляет собой голосового винного ассистента и цифрового сомелье. Пользователь может разговаривать с ассистентом на естественном языке, задавать вопросы о винодельнях, винах, сортах винограда, регионах, гастрономических сочетаниях, производстве вина и винном туризме.

Основной интерфейс может работать:

- на сайте;
- на вертикальном экране в магазине, ресторане или винодельне;
- на выставке;
- в отеле или туристическом центре;
- как отдельная брендированная версия для конкретной винодельни.

В интерфейсе может присутствовать анимированный персонаж или видеоаватар сомелье. Сейчас требуется изучить возможность добавить динамический B-roll: визуальные материалы, которые автоматически появляются во время ответа ассистента и иллюстрируют то, о чём он говорит.

Пример:

Ассистент говорит:
«Это вино производится из сорта Fetească Neagră. Виноград выращивают в регионе Ștefan Vodă, а после ферментации вино выдерживают в дубовых бочках».

В этот момент интерфейс может последовательно показать:

1. бутылку конкретного вина;
2. виноград Fetească Neagră;
3. карту региона Ștefan Vodă;
4. виноградники винодельни;
5. кадры дубовых бочек;
6. карточку с характеристиками вина;
7. кнопку или QR-код «Где купить».

Визуальные материалы должны быть синхронизированы со смыслом ответа, но на первом этапе не требуется точная покадровая синхронизация уровня профессионального видеомонтажа.

## Главная задача

Провести технический аудит проекта и определить, можно ли добавить в WINE AI систему динамического B-roll без разрушения текущей realtime-архитектуры голосового ассистента.

После аудита подготовить подробный пошаговый план реализации.

На этом этапе запрещено начинать крупную реализацию без отдельного подтверждения. Допускаются только безопасные исследования кода, схемы, небольшие proof-of-concept проверки и предложения интерфейсов.

## Часть 2. Оценить возможность реализации

После изучения проекта дай честную техническую оценку.

Нужно ответить:

### 2.1. Реализуемость

Можно ли внедрить динамический B-roll в текущую архитектуру?

Укажи:

- что уже есть в проекте и может быть переиспользовано;
- каких компонентов не хватает;
- какие части потребуют изменения;
- какие риски могут повредить realtime-диалогу;
- что можно сделать быстро;
- что окажется сложным и дорогим.

Оцени отдельно:

1. простой вариант;
2. средний вариант;
3. продвинутый вариант.

### 2.2. Простой вариант

Ассистент завершает формирование ответа или его смыслового блока, после чего frontend получает несколько визуальных команд.

Пример:

```json
{
  "type": "visual_sequence",
  "items": [
    {
      "visualType": "wine_bottle",
      "entityType": "wine",
      "entityId": "wine_123",
      "durationMs": 5000
    },
    {
      "visualType": "vineyard_video",
      "entityType": "winery",
      "entityId": "winery_45",
      "durationMs": 7000
    }
  ]
}
```

Frontend показывает материалы по очереди.

Проверь, является ли такой подход самым разумным для MVP.

### 2.3. Средний вариант

Ответ ассистента разбивается на смысловые блоки. Для каждого блока формируется visual cue.

Пример:

```json
{
  "responseId": "response_789",
  "segments": [
    {
      "segmentId": "segment_1",
      "text": "Это вино создано из сорта Fetească Neagră.",
      "visualCue": {
        "visualType": "grape_variety",
        "entityId": "feteasca_neagra"
      }
    },
    {
      "segmentId": "segment_2",
      "text": "Виноград выращивают в регионе Ștefan Vodă.",
      "visualCue": {
        "visualType": "region_map",
        "entityId": "stefan_voda"
      }
    }
  ]
}
```

Проверь, можно ли связать смену визуальных материалов с воспроизведением каждого текстового или аудиоблока.

### 2.4. Продвинутый вариант

Система получает timestamps отдельных слов, фраз или аудиосегментов и синхронизирует B-roll почти как монтажная программа.

Проверь:

- предоставляет ли текущий voice provider timestamps;
- можно ли получить word-level timestamps;
- можно ли получить sentence-level timestamps;
- потребуется ли отдельный forced alignment;
- насколько это увеличит задержку;
- насколько такая точность оправдана для WINE AI.

Не предлагай этот вариант как MVP, если его цена и сложность не оправданы.

## Часть 3. Предложить целевую архитектуру

Подготовь рекомендуемую архитектуру системы.

Она должна содержать следующие логические компоненты.

### 3.1. Visual Cue Generator

Компонент получает:

- текст ответа;
- найденные KOS-сущности;
- текущий контекст разговора;
- идентификатор винодельни;
- идентификатор вина;
- язык ответа;
- доступные медиа.

На выходе компонент возвращает структурированный список visual cues.

Он не должен генерировать произвольные URL. Он должен ссылаться на проверенные сущности и медиа из базы.

Пример:

```ts
type VisualCue = {
  id: string;
  responseId: string;
  sequence: number;
  cueType:
    | "wine_bottle"
    | "winery_logo"
    | "vineyard_photo"
    | "vineyard_video"
    | "cellar_video"
    | "grape_variety"
    | "region_map"
    | "wine_card"
    | "aroma_card"
    | "food_pairing"
    | "winery_history"
    | "purchase_cta";
  entityType?: "wine" | "winery" | "region" | "grape" | "food";
  entityId?: string;
  mediaAssetId?: string;
  startCondition?: "immediate" | "after_previous" | "segment_start";
  durationMs?: number;
  priority?: number;
  fallbackCueType?: string;
};
```

Проверь и улучши контракт. Не усложняй его без необходимости.

### 3.2. Media Registry

Нужен единый реестр медиа.

Он должен связывать материалы с сущностями:

- винодельня;
- вино;
- винтаж;
- регион;
- сорт винограда;
- гастрономическая пара;
- общая винная тема.

Предложи минимальную схему данных.

Возможный вариант:

```ts
type MediaAsset = {
  id: string;
  entityType: string;
  entityId: string;
  mediaType: "image" | "video" | "animation";
  category: string;
  url: string;
  thumbnailUrl?: string;
  orientation: "portrait" | "landscape" | "square";
  language?: string;
  durationMs?: number;
  source?: string;
  copyrightStatus?: string;
  isApproved: boolean;
  priority: number;
  metadata?: Record<string, unknown>;
};
```

Проверь, где это лучше хранить:

- в существующем KOS;
- в отдельных таблицах Postgres;
- в JSON на этапе прототипа;
- в Cloudflare R2 вместе с метаданными в Postgres;
- в другом уже используемом хранилище.

### 3.3. Visual Orchestrator

Frontend-компонент должен:

- получать visual cues;
- загружать медиа заранее;
- показывать материалы по очереди;
- обрабатывать переходы;
- не останавливать голос при ошибке загрузки;
- не ломать аватар;
- отменять старую очередь при новом ответе;
- очищать состояние при прерывании ответа;
- корректно работать при barge-in;
- поддерживать fallback;
- учитывать вертикальный и горизонтальный экран.

Особенно проверь lifecycle:

```
idle
listening
user_speaking
thinking
assistant_speaking
interrupted
completed
error
```

B-roll должен подчиняться lifecycle ответа, а не жить отдельным несвязанным таймером.

### 3.4. Media Resolver

Компонент должен находить лучший медиафайл для visual cue.

Пример логики:

1. найти точное медиа конкретного вина;
2. если его нет, взять медиа винодельни;
3. затем медиа региона;
4. затем тематическое винное медиа;
5. затем нейтральный fallback;
6. если ничего нет, оставить аватар без B-roll.

Система не должна показывать случайную чужую бутылку вместо нужного вина.

### 3.5. Preloader и Cache

Проверь, как лучше:

- загружать первые материалы до начала речи;
- предзагружать следующий ролик;
- кешировать изображения и короткие видео;
- ограничивать размер видео;
- выбирать подходящий формат;
- не тратить мобильный трафик без контроля;
- работать при медленном интернете;
- работать на вертикальном экране с мини-ПК.

## Часть 4. Определить, откуда берутся visual cues

Сравни несколько подходов.

### Подход A. LLM возвращает текст и cues одновременно

Пример:

```json
{
  "spokenText": "Это вино производится...",
  "visualCues": []
}
```

Плюсы и минусы:

- простота;
- риск повреждения формата;
- задержка;
- зависимость от модели;
- сложность при streaming.

### Подход B. Отдельный быстрый post-processing

Основная модель генерирует ответ. После получения текста маленькая модель или rule-based слой формирует cues.

Проверь:

- можно ли запускать это параллельно с TTS;
- какая будет задержка;
- можно ли обрабатывать ответ частями;
- подходит ли небольшая быстрая модель;
- можно ли сначала применить правила, затем LLM fallback.

### Подход C. Cues строятся из RAG/KOS retrieval

Если при ответе уже были найдены сущности, visual cues создаются непосредственно из этих сущностей.

Пример:

```
retrieved wine_id -> wine bottle
retrieved winery_id -> vineyard media
retrieved region_id -> region map
retrieved grape_id -> grape card
```

Этот подход предпочтителен там, где он даёт детерминированный результат.

### Рекомендуемый гибрид

Проверь целесообразность следующей схемы:

1. KOS/RAG возвращает сущности;
2. rule-based слой выбирает очевидные visual cues;
3. маленькая модель определяет порядок и смысловые переходы;
4. Media Resolver подбирает конкретные файлы;
5. frontend показывает очередь.

Дай своё заключение, какой подход лучше для текущего проекта.

## Часть 5. Синхронизация с голосом

Изучи текущий аудиопайплайн и предложи практический способ синхронизации.

Рассмотри четыре уровня.

**Уровень 1. Таймеры** — B-roll меняется каждые 4–8 секунд. Это допустимо для первого прототипа.

**Уровень 2. Аудиочанки** — каждый аудиочанк или абзац связан со своим visual cue.

**Уровень 3. События предложений** — TTS создаётся отдельными предложениями. Перед воспроизведением каждого предложения frontend включает соответствующий cue.

**Уровень 4. Word timestamps** — используются точные временные метки.

Для каждого уровня укажи:

- сложность;
- задержку;
- надёжность;
- требования к текущему voice provider;
- пользу для пользователя;
- пригодность для MVP.

Предложи конкретный уровень для первой версии.

## Часть 6. Работа с прерыванием речи

Пользователь может перебить ассистента.

Проверь, как система должна вести себя при barge-in:

- остановить текущую речь;
- отменить оставшиеся visual cues;
- остановить или скрыть текущее видео;
- не показывать cues от старого ответа;
- очистить responseId;
- начать новую очередь только для нового ответа.

Нужно использовать responseId, sessionId или turnId, чтобы старые события не попадали в новый ответ.

Пример проверки:

```ts
if (cue.responseId !== activeResponseId) {
  return;
}
```

Посмотри, какие идентификаторы уже существуют в проекте, и переиспользуй их.

## Часть 7. Интерфейс

Предложи несколько режимов отображения.

**Режим A. Avatar First** — аватар остаётся главным. B-roll появляется рядом, за аватаром, в отдельной карточке или как временная вставка.

**Режим B. B-roll First** — во время рассказа медиа занимает почти весь экран, а аватар уменьшается до небольшого окна.

**Режим C. Split Screen** — экран делится на две части: сомелье и визуальный материал.

**Режим D. Full-screen Inserts** — на несколько секунд B-roll занимает весь экран, затем интерфейс возвращается к персонажу.

Для MVP предложи один основной режим. Учти:

- вертикальный экран;
- мобильный телефон;
- desktop;
- kiosk mode;
- текущий дизайн WINE AI.

Не начинай редизайн всего интерфейса.

## Часть 13. Proof of Concept

Предложи минимальный PoC.

PoC должен работать только для одной демо-винодельни и нескольких вин.

Например:

- одна винодельня;
- три вина;
- одно видео виноградника;
- одно видео погреба;
- изображения бутылок;
- карта региона;
- карточка сорта винограда;
- карточка гастропары.

PoC должен подтвердить:

- visual cues можно создать;
- медиа можно связать с сущностями;
- frontend может менять материалы во время речи;
- прерывание очищает очередь;
- ошибки B-roll не ломают голос;
- задержка остаётся приемлемой.

Не создавай генератор видео, сложный монтажный движок или ML-систему распознавания сцен.

## Часть 14. Рекомендуемый порядок реализации

После аудита предложи этапы.

Ожидаемая логика этапов:

**Этап 0. Аудит** — изучить архитектуру; найти точки интеграции; описать риски; подтвердить или отклонить предложенную схему.

**Этап 1. Статический прототип** — создать несколько локальных media assets; вручную задать visual cues; реализовать frontend visual queue; проверить lifecycle и barge-in.

**Этап 2. Media Registry** — добавить таблицу или временный JSON-реестр; связать медиа с KOS-сущностями; реализовать Media Resolver.

**Этап 3. Автоматические cues** — использовать найденные KOS-сущности; добавить rule-based Visual Cue Generator; не подключать LLM без необходимости.

**Этап 4. Смысловая последовательность** — добавить небольшую модель или structured output; определить порядок кадров; ограничить число cues; убрать повторы.

**Этап 5. Dashboard** — загрузка медиа; привязка к сущностям; preview; одобрение; включение и отключение.

**Этап 6. Улучшенная синхронизация** — sentence-level cues; связь с аудиочанками; более точные переходы.

Для каждого этапа оцени:

- какие файлы будут изменены;
- какие новые файлы появятся;
- зависимости;
- риски;
- тесты;
- критерии готовности.

## Часть 15. Что нужно получить в результате аудита

Подготовь отчёт со следующей структурой.

1. **Executive summary** — кратко: реализуемо ли; лучший вариант для MVP; главные риски; примерный объём изменений.
2. **Current architecture findings** — опиши фактическую архитектуру проекта со ссылками на файлы и модули.
3. **Recommended architecture** — дай схему компонентов и поток данных.

   Пример потока:

   ```
   User question
   -> Realtime conversation service
   -> RAG/KOS retrieval
   -> Assistant response
   -> Visual Cue Generator
   -> Media Resolver
   -> Visual Queue
   -> Avatar + B-roll renderer
   ```

4. **Integration points** — перечисли точные файлы, компоненты, hooks, services и API routes, которые придётся изменить.
5. **Data model** — предложи минимальную схему таблиц или TypeScript-типов.
6. **Realtime lifecycle** — опиши состояния, responseId, cancellation и barge-in.
7. **Implementation plan** — дай пошаговый план от PoC до production.
8. **Risks** — раздели риски на: архитектурные; realtime; frontend; media storage; производительность; контент; авторские права; эксплуатацию.
9. **Alternatives rejected** — объясни, какие решения не стоит использовать сейчас и почему.
10. **Final recommendation** — сформулируй конкретную рекомендацию: что делать первым; что отложить; какой вариант синхронизации выбрать; какой формат B-roll использовать; сколько сущностей включить в PoC.

## Ограничения

1. Не переписывай realtime-ядро без доказанной необходимости.
2. Не добавляй новую сложную инфраструктуру только ради B-roll.
3. Не привязывайся к одному LLM-провайдеру без необходимости.
4. Не генерируй визуальные URL прямо из ответа модели.
5. Не показывай медиа, не связанное с проверенной сущностью.
6. Не допускай, чтобы ошибка B-roll останавливала голосовой диалог.
7. Не запускай тяжёлые video pipelines на сервере без доказанной пользы.
8. Не внедряй word-level synchronization в MVP.
9. Не создавай универсальную видеоплатформу. Задача ограничена WINE AI.
10. Сохраняй существующие архитектурные границы проекта.
11. Учитывай текущий этап продукта и пилот с ограниченным числом виноделен.
12. Перед изменениями проверь AGENTS.md и локальные инструкции репозитория.
13. Не трогай секреты, production-данные и несвязанные части проекта.
14. Не делай автоматический commit, push или deploy без отдельной команды.

## Главный инженерный принцип

B-roll является необязательным визуальным слоем поверх существующего диалога.

Голосовой ассистент должен продолжать работать, даже если:

- visual cue не был создан;
- медиа отсутствует;
- видео не загрузилось;
- CDN недоступен;
- frontend не поддерживает видео;
- пользователь перебил ответ.

Архитектура должна быть построена так, чтобы B-roll можно было отключить feature flag без изменения основного realtime-пайплайна.

## Формат первого ответа

На первом проходе не вноси крупные изменения.

Сначала выдай:

1. что ты изучил;
2. какие файлы и модули относятся к задаче;
3. реализуема ли схема;
4. какой вариант предлагаешь для MVP;
5. где находятся главные риски;
6. какой будет пошаговый план;
7. какие вопросы остались после изучения кода;
8. какой минимальный PoC ты предлагаешь;
9. какие изменения ты готов сделать на следующем шаге.

После этого остановись и дождись отдельной команды на реализацию.

## Примечание (практический выбор автора задания)

Практический выбор для первого прототипа: KOS-сущности плюс простой rule-based генератор cues, очередь на frontend и смена кадров по предложениям или таймерам. Точную синхронизацию по словам пока лучше не трогать — она съест много времени, а визуально даст меньше пользы, чем хорошие официальные фото, короткие ролики и корректная отмена кадров при прерывании ассистента.
