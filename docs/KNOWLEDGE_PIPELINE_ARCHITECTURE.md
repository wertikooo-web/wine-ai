# Wine Knowledge Pipeline — анализ и архитектура

Статус: **утверждено пользователем 2026-07-19, реализация начата.** Раздел 13 ниже — уточнение архитектуры по рекомендациям из внешнего анализа (GPT), включённое перед стартом реализации.

---

## 1. Текущее состояние (что уже есть)

`src/knowledge/` сейчас — простой, но честно спроектированный **статический** конвейер:

```
knowledge/source/*.{md,txt,json,csv}   — вручную написанные документы с frontmatter-метаданными
        ↓ src/knowledge/loader.js       — парсинг frontmatter + чанкинг по абзацам
        ↓ src/knowledge/index.js        — buildIndex()/loadIndex(), пишет knowledge/index/index.json
        ↓ src/knowledge/search.js       — keyword/term-overlap скоринг (без embeddings)
        ↓ src/tools/searchWineKnowledge.js (и 4 других tool-а) — вызываются моделью через function calling
```

Ключевое свойство, которое нужно сохранить: **контракт `search(query, options) → {hits}` уже отделён от источника данных**. Инструменты (`src/tools/*.js`) не знают, что индекс — плоский JSON-файл с keyword-скорингом. Это и есть точка расширения — новый Knowledge Pipeline должен продолжать отвечать на этот же контракт, просто наполняя `knowledge/index/` (или его преемника) намного более богатыми данными и, позже, отдавая эту же функцию через embeddings/векторную БД вместо term-overlap.

**Explicit non-goal текущей системы:** она не собирает данные сама — всё пишется руками в `knowledge/source/`. Именно это и должен закрыть Knowledge Pipeline.

## 2. Предлагаемая архитектура

Отдельный сервис/модуль `knowledge-pipeline/` — **физически отдельный от `wine-ai-realtime`'s runtime**, тем же принципом независимости, что описан в `AGENTS.md` и `docs/WINE_AI_MIGRATION_PLAN.md`: голосовой сервис не должен зависеть от пайплайна во время выполнения, а пайплайн не должен требовать переустановки голосового сервиса, чтобы работать.

```
knowledge-pipeline/                  (отдельный процесс/репозиторий или подпапка с изолированным package.json)
├── config/
│   ├── sources.yaml                 — реестр источников (см. §3)
│   └── trust-levels.yaml            — веса A/B/C/D (см. §5)
├── sources/                          — snapshot конфигурации по конкретным сайтам/винодельням
├── discovery/                        — поиск НОВЫХ источников (§6)
├── crawler/                          — скачивание HTML/PDF по известным источникам
├── parser/                           — HTML/PDF → чистый текст + структура
├── extractor/                        — текст → факты по схеме (винодельня/сорт/награда/…)
├── deduplicator/                     — многоуровневый дедуп (§7)
├── verifier/                         — сверка фактов, конфликты между источниками, trust-level арбитраж
├── embeddings/                       — интерфейс генерации векторов (провайдер-агностичный)
├── indexer/                          — запись в векторную БД + метаданные (интерфейс, не конкретная БД)
├── scheduler/                        — периодический запуск всего цикла
├── reports/                          — machine- и human-readable отчёты о каждом прогоне
└── knowledge-base/                   — САМИ ДАННЫЕ (см. §4), это то, что читает wine-ai-realtime
```

Граница с `wine-ai-realtime`: пайплайн пишет в `knowledge-base/` (файлы + векторный индекс); `wine-ai-realtime` только читает оттуда через тот же `search()`-контракт. Никакого прямого вызова пайплайна из голосового сервиса в реальном времени — только асинхронное обновление данных на диске/в БД.

### 2.1 Модули и интерфейсы (сигнатуры, не реализация)

```js
// discovery/ — ищет НОВЫЕ источники, а не только обходит известные
discoverSources({ queries: string[], trustLevelFloor }) => Promise<CandidateSource[]>

// crawler/ — скачивает контент по известному источнику
fetchDocument(source: Source) => Promise<{ url, contentType, rawBytes, fetchedAt }>

// parser/ — сырые байты → чистый текст + базовая структура (заголовки, таблицы)
parse(raw: RawDocument) => Promise<ParsedDocument>  // { text, headings[], tables[], images[] }

// extractor/ — текст → факты по строгой схеме, ПО КАТЕГОРИИ документа
extractFacts(parsed: ParsedDocument, schema: WineryCardSchema | GrapeSchema | AwardSchema | ...) => Promise<ExtractedFacts>

// verifier/ — межисточниковая сверка + конфликты
verifyFacts(facts: ExtractedFacts, existingKnowledge) => Promise<{ accepted, conflicts, needsReview }>

// deduplicator/ — несколько уровней (§7)
deduplicate(candidate: Document, existing: Document[]) => Promise<{ isDuplicate, matchedId, similarity }>

// embeddings/ — провайдер-агностичный интерфейс
embed(texts: string[]) => Promise<number[][]>   // конкретная модель — деталь реализации за интерфейсом

// indexer/ — тоже провайдер-агностичный (см. §8 про Qdrant)
upsert(chunks: {id, text, vector, metadata}[]) => Promise<void>
search(queryVector, filters) => Promise<{id, score, metadata}[]>

// scheduler/ — независимый процесс, будит весь цикл
runPipelineCycle() => Promise<PipelineReport>
```

Все интерфейсы — простые async-функции с явными входом/выходом, без скрытого состояния (тот же принцип, что уже применён в `src/tools/toolHelpers.js`: логирование времени выполнения, структурированные ошибки, никаких утечек внутренних деталей наружу).

## 3. Источники и Trust Levels

Единый реестр `config/sources.yaml`, а не разбросанный по коду список сайтов:

```yaml
- id: onvv-official
  type: official
  trust: A
  base_url: https://...
  categories: [wineries, regions, grapes, tourism, events, news]
- id: winery:purcari
  type: winery
  trust: A
  base_url: https://...
- id: decanter
  type: international_media
  trust: B
  usage: reference_only        # используется только как доп. контекст, не как основной факт
- id: vivino
  type: aggregator
  trust: C
  fields_allowed: [avg_rating, style, descriptors]   # отзывы НЕ используются как факты
```

`trust` (A/B/C/D) — обязательное поле метаданных на каждом чанке в итоговой базе (расширение уже существующего `confidence` в `src/knowledge/loader.js`). Модель должна получать эту метку вместе с фрагментом и обязана честно понижать уверенность ответа для B/C/D-источников — это уже частично заложено в персону (`src/persona/wineExpertPersona.js`: «различай факт, мнение и рекомендацию»), пайплайн просто даёт модели материал для этого различения.

## 5. Дедупликация (многоуровневая)

1. **URL-уровень** — уже видели этот URL с тем же content-hash → пропуск.
2. **Точный текст** — hash нормализованного текста (уже есть паттерн в `src/knowledge/loader.js`'s `stableId`, расширяется).
3. **Почти дубликат** — similarity по embeddings (косинусное сходство выше порога) или shingling/MinHash — не требует LLM-вызова на каждую пару.
4. **Одно и то же событие из разных источников** — сопоставление по структурированным полям (винодельня+дата+тип события), не по тексту — тут работает `verifier/`, а не `deduplicator/`.

---

## 6. Discovery (поиск новых источников)

Отдельный от `crawler/` модуль — `crawler/` умеет обходить ТОЛЬКО известные источники из `sources.yaml`; `discovery/` ищет НОВОЕ через поисковые запросы (`Moldovan wine`, `винодельни Молдовы`, `Wine Tourism Moldova`, …) и предлагает кандидатов в реестр источников **для ручного/полуавтоматического подтверждения**, а не для мгновенного автодобавления — это важный контроль качества: discovery не должен сам себе назначать trust level A.

---

## 7. Версионирование

Каждое обновление документа создаёт новую версию в `_versions/`, а не перезаписывает файл:

```
wineries/Purcari/_versions/
  2026-07-18T10-00-00Z.md
  2026-08-01T09-00-00Z.md
```

Текущий `history.md` — всегда последняя версия; история изменений читается из `_versions/` + отдельного `changelog.jsonl` (source, что изменилось, когда).

---

## 8. Embeddings и векторный индекс — контракт, не привязка к вендору

`embeddings/` и `indexer/` — оба за интерфейсом (`embed()`/`upsert()`/`search()` из §2.1), чтобы модель embeddings и векторная БД менялись независимо друг от друга и от остального кода. Qdrant — разумный дефолт (self-hosted, простой REST/gRPC API, есть managed cloud-тариф), но `indexer/` не должен течь Qdrant-специфику наружу — тот же принцип, что уже применён к realtime-провайдеру (`src/realtime/geminiLiveProvider.js` спрятан за нейтральным контрактом в `realtimeServer.js`).

**Точка интеграции с текущим RAG**: `src/knowledge/search.js`'s `search(query, options)` — сохраняется как публичный контракт для `src/tools/*`. Когда появится векторный слой, `search.js` внутри переключается на `indexer.search()` вместо term-overlap, но сигнатура и вызывающий код (`src/tools/searchWineKnowledge.js` и другие) не меняются вообще. Term-overlap (текущая реализация) остаётся как fallback для пустой/маленькой базы, где embeddings не оправданы.

## 9. Точки интеграции с Dashboard

Новый раздел **Knowledge** (не путать с уже существующей вкладкой "Knowledge base" в `public/dashboard.html` — она расширяется, а не дублируется):

- Последнее обновление / следующий запуск (из `scheduler/`)
- Источников проверено / новых документов / обновлённых / ошибок (из `reports/`)
- Статус индексации, размер базы, размер векторного индекса
- Кнопки: «Запустить обновление вручную», «Переиндексировать», «Посмотреть лог», «Посмотреть найденные изменения»

Технически: пайплайн пишет `reports/latest.json`; `src/server.js` получает пару новых read-only эндпоинтов (`/api/knowledge/pipeline-status`, `/api/knowledge/pipeline-log`), которые просто читают этот файл/директорию — **никакого прямого вызова пайплайна из голосового сервиса**, только чтение уже готовых отчётов. Ручной запуск («Запустить обновление вручную») дергает пайплайн как отдельный процесс (аналогично тому, как `POST /api/knowledge/reindex` сейчас дергает `buildIndex()`), не блокируя realtime-сессии.

## 10. Риски (важно проговорить до реализации)

1. **Юридический/этический риск скрапинга.** Decanter, Wine Spectator, Wine Enthusiast, Wine Searcher — у большинства платный контент и явные ограничения в Terms of Service на автоматический сбор. Vivino API формально ограничен. Перед подключением каждого B/C-источника нужно явно проверить ToS/robots.txt — иначе рискуем блокировкой IP или юридической претензией. Официальный сайт ONVV (Wine of Moldova) и сайты самих виноделен — наименьший риск (это их собственный маркетинговый контент, обычно приветствуется).
2. **Vivino — отзывы пользователей нельзя использовать как факты** (уже зафиксировано в задании) — это должно быть жёстким программным ограничением в `extractor/`, а не просто договорённостью в документации.
3. **Стоимость.** Embeddings-вызовы + hosted-векторная БД + периодический краулинг — это реальные, растущие расходы, в отличие от текущего keyword-based RAG (бесплатно). Нужно заложить бюджет и лимиты (например, троттлинг discovery, кэширование embeddings по content-hash, чтобы не пересчитывать вектора для неизменившегося текста).
4. **Rate-limiting и вежливый краулинг.** Обязательны: User-Agent с контактом, задержки между запросами, уважение robots.txt, экспоненциальный backoff.
5. **Конфликт данных между источниками** — два сайта могут указывать разные годы основания винодельни. `verifier/` должен явно помечать конфликт и выбирать по trust level, а не молча брать последнее скачанное значение.
6. **Масштаб инфраструктуры на Railway.** Текущий деплой — один лёгкий Node-процесс. Краулер+embeddings+векторная БД — это либо отдельный Railway-сервис (рекомендуется, чтобы не грузить realtime-инстанс), либо вообще отдельная инфраструктура. Нужно решить это явно на этапе MVP, не постфактум.

## 11. Поэтапный план (MVP → Production)

**Этап 0 (уже есть, не трогаем).** Текущий keyword-based `src/knowledge/*` продолжает работать как fallback весь путь — ни один этап ниже не должен его сломать.

**Этап 1 — Реестр источников + карточка винодельни (без краулинга).**
`config/sources.yaml` с 3–5 винодельнями + ONVV, вручную заполненные `card.json` по предложенной схеме, конвертер `card.json` → существующий frontmatter-формат `knowledge/source/*.md`, чтобы уже сейчас улучшить текущую базу без нового кода краулинга.

**Этап 2 — Crawler + Parser для официальных источников (trust A only).**
Только ONVV + сайты самих виноделен. Никаких discovery, никаких B/C/D источников на этом этапе — минимизируем юридический риск, пока процесс не обкатан.

**Этап 3 — Extractor + Deduplicator (уровни 1–2: URL/точный текст).**
Факты по схемам (винодельня/сорт/регион), сохранение в `knowledge-base/` структуре из §4, версионирование (§9) сразу закладывается, не откладывается на потом.

**Этап 4 — Embeddings + indexer за нейтральным контрактом, `search.js` переключается на них при непустом векторном индексе.**
Выбор конкретной embeddings-модели и Qdrant — на этом этапе, не раньше (нет смысла проектировать вокруг конкретного вендора заранее).

**Этап 5 — Scheduler + Dashboard-раздел Knowledge (статус, ручной запуск, логи).**

**Этап 6 — Discovery + B/C источники (Decanter/Wine Searcher/Vivino с ограничениями) + Awards + Verifier для межисточниковых конфликтов.**
Самый рискованный этап юридически — начинать только после явного решения по каждому источнику отдельно.

## 12. Критерии готовности MVP (конец этапа 3)

- `knowledge-base/wineries/<Name>/card.json` существует для ≥10 виноделен, автоматически собранных с их официальных сайтов + ONVV.
- Дедупликация уровня 1–2 отсекает повторные краулинги без ручного вмешательства.
- Версии сохраняются, `changelog.jsonl` читаем.
- `src/knowledge/search.js` не изменил свой публичный контракт; `src/tools/*` не тронуты.
- Отчёт о прогоне (`reports/latest.json`) содержит: источников проверено, новых/обновлённых документов, ошибок.
- Ничего из текущего `wine-ai-realtime` runtime (WS, turn state machine, realtime-провайдер) не изменено.

## 13. Уточнение по рекомендациям (GPT-анализ, 2026-07-19)

Пользователь принёс независимый анализ той же задачи с рекомендациями, которые уточняют (не отменяют) план выше. Ключевые идеи, которые приняты и встроены в реализацию v1:

### 13.1 Два слоя вместо одного

- **Подтверждённая база знаний** (`knowledge/source/` + индекс, уже существует) — то, что реально используется в ответах ассистента.
- **Лента обнаруженных материалов** (новое, `knowledge/discovered/`) — всё, что нашёл краулер, со статусом `pending`/`approved`/`rejected`. В подтверждённую базу попадает только `approved`.

Это прямая защита от превращения базы в свалку рекламных текстов/устаревших цен — именно риск, который уже был отмечен в §10 выше.

### 13.2 Trust-level автоматизация публикации

- **Trust A (официальный сайт ONVV/винодельни/конкурса)** — можно автоматически переводить в `approved` и подключать к RAG.
- **Trust B (СМИ)** — попадает в новостную ленту, не в фактическую базу автоматически.
- **Trust C (блоги/агрегаторы/магазины)** — только как "найдено", никогда не как факт без ручного подтверждения.

Отдельное жёсткое правило (встроено в verifier, не просто как договорённость): цены, часы работы, наличие, контакты, винтажи, награды, владельцы — **никогда не обновляются автоматически** без пометки "требует проверки", даже если источник trust A.

### 13.3 Дедупликация — реализуем 2 из 3 уровней в v1

1. Точный URL (`content_hash` по URL) — v1.
2. Хеш очищенного текста — v1.
3. Семантическое совпадение (заголовок/дата/участники через embeddings) — отложено до этапа 4 (нужны embeddings).

### 13.4 Хранилище — файловое в v1, не Postgres/Qdrant

GPT предлагает Postgres (структурированные данные: sources/documents/news/wineries/wines/awards/crawl_runs) + Qdrant (embeddings). Это правильная архитектура для Production, но для v1 добавлять два внешних управляемых сервиса — необоснованный риск и стоимость на этом шаге (см. §10, риск 3 и 6). **Решение**: v1 использует JSON-файлы (`knowledge/discovered/*.json`, тот же паттерн, что уже работает в `src/knowledge/index.js`) со **строго той же схемой полей**, что предложена для Postgres/Qdrant payload — переход на реальные БД позже станет заменой хранилища, а не переписыванием структуры данных или API.

Схема записи (файловый аналог Postgres `documents`/`news` + Qdrant payload, объединены):

```json
{
  "id": "doc_...",
  "title": "...",
  "url": "https://...",
  "publisher": "Wine of Moldova",
  "publishedAt": null,
  "fetchedAt": "2026-07-19T02:10:00Z",
  "language": "ro",
  "sourceId": "onvv-official",
  "trustLevel": "A",
  "contentHash": "sha256...",
  "topics": ["winery", "region", "news"],
  "entities": { "wineries": [], "wines": [], "grapes": [], "regions": [] },
  "summary": "...",

  "text": "...",
```

### 13.5 Discovery (веб-поиск) — этап 2, не в v1

Автоматический поиск по интернету (через Firecrawl Search или аналог) требует стороннего API-ключа, который пользователь должен сам завести и вписать (см. правило про секреты в §10/AGENTS.md) — это явно откладывается на следующий этап, а не блокирует v1. v1 работает только по заранее заданному, вручную проверенному реестру источников.

### 13.6 Cron на Railway

Рекомендация "раз в 3 дня" реализуется как ежедневный cron (`0 2 * * *`) с внутренней проверкой "прошло ли ≥72 часов с последнего успешного запуска" — так расписание не ломается на границе месяца (ровно как предложено в анализе). Отдельный Railway-сервис из того же репозитория, команда `npm run knowledge:update`, cron-триггер вместо постоянно работающего процесса.

### 13.7 Dashboard: Knowledge Monitor

Новая панель (расширяет уже существующую вкладку "Knowledge base"): счётчики (проверено источников/новых материалов/дубликатов/ошибок), три вкладки **Approved / Pending / Rejected**, на каждой карточке — заголовок, источник, дата, сущности, уровень доверия, кнопки «Открыть оригинал» / «Одобрить» / «Отклонить» / «Переиндексировать».

### 13.8 Первая версия (то, что реализуется сейчас)

- Реестр источников: **wineofmoldova.com** (ONVV, единственный подтверждённый на данный момент официальный домен — см. ниже) на 3 языках (ro/ru/en), разделы: About ONVV, Our Wineries, Wine Tourism, Moldovan Wine, Wine Regions, News & Media.
- Краулер на `fetch()` + `cheerio` (без Firecrawl — не требует API-ключа).
- Дедуп уровня 1–2.
- Очередь `knowledge/discovered/` со статусами, trust-A автоматически помечается `approved`, но факты о ценах/наградах/владельцах внутри всё равно помечаются `needs_review` даже при статусе `approved` документа в целом.
- `approved` документы конвертируются в существующий frontmatter-формат `knowledge/source/*.md` и переиндексируются через уже работающий `buildIndex()` — без изменения контракта `search()`.
- `npm run knowledge:update` — команда запуска всего цикла.
- Dashboard Knowledge Monitor — работает поверх файлового хранилища.
- Postgres/Qdrant/Firecrawl/дискавери — осознанно отложены на следующий этап (§13.5), схема данных уже с ними совместима.

**Важно про источники:** индивидуальные сайты конкретных виноделен (Purcari, Cricova, Castel Mimi и т.д.) **не были проверены браузером** и не добавлены в реестр v1, чтобы не рисковать вписыванием неверных/несуществующих URL. `wineofmoldova.com` подтверждён напрямую (реальные ссылки на `/en/our-wineries/`, `/en/wine-regions/`, `/en/news-and-media/` и т.д.) и уже агрегирует информацию по всем винодельням, регионам и новостям — этого достаточно для честного v1. Добавление прямых сайтов виноделен — следующий шаг, каждый URL нужно будет так же подтвердить перед добавлением в реестр.
